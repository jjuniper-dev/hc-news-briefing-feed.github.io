Health Canada / PHAC Enterprise AI

PATH Agent Briefing

Protected AI Technology Hub (PATH) is the target-state enterprise AI control plane for governed execution, policy enforcement, and defensible scale.

Critical distinction: PATH is not a data platform, lab, or toolset. It is the operating system for AI across the enterprise.

1) Core Definition + Why PATH Exists

What PATH provides

  • Governed AI execution
  • Standardized deployment patterns
  • End-to-end auditability
  • Secure, policy-enforced model/data access

Observed current-state blockers

  • Fragmented AI patterns and incomplete delivery paths
  • No standard ingress/networking path
  • Heavy cross-team dependency for each deployment
  • No unified runtime and governance model
Conclusion: The organization is scaling data capabilities, but not AI execution control and governance at enterprise scale.

2) Core Architectural Principle

Platforms consume governance; they do not define it.

LayerRole
Data PlatformData capability

3) Canonical PATH Architecture: Three Planes + Reuse

A. Control Plane (PATH Core)

  • Identity & access: Microsoft Entra ID, RBAC, PIM, project-level isolation.
  • AI Gateway (critical): Azure API Management as the mandatory gateway for model access, metering, rate limiting, request validation, and policy controls.
  • Governance: Azure Policy, TBS AIA/PIA gates, classification enforcement (Unclassified to Protected B), prompt policy controls.
  • Observability: Azure Monitor + Log Analytics capture prompt/response/user/model/tokens/time for every request.
  • Data governance integration: Purview classification, lineage, and access inheritance.
  • FinOps: Token tracking, project budget controls, and enterprise cost visibility.

B. Runtime Plane (Execution Layer)

  • Project/program-isolated Azure subscriptions.
  • Foundry workspace per project with Azure AI Search, Functions/Logic Apps, ADLS Gen2, private networking, and App Insights.
  • Non-negotiable rule: no direct model access; all requests route through PATH gateway.

C. Data Integration Layer

  • Connects enterprise data platform (Fabric/Databricks) and internal/external data sources.
  • Uses governed RAG pipelines, data APIs, and controlled ingestion.
  • Enforced by Purview + Entra ID to preserve trusted data lineage.

D. Reuse Layer (Acceleration)

  • Shared model catalog, prompt registry, RAG templates, and pre-approved pipelines.
  • Integration with Power Platform and M365 Copilot APIs for delivery speed and consistency.

4) Operating Model and Governance

Onboarding target: under 5 days

  1. Intake request
  2. Subscription provisioning
  3. Automated policy + networking baseline
  4. Foundry workspace creation
  5. Gateway entitlement and route provisioning
  6. Budget + monitoring activation

Production gates (CI/CD)

  • Security review
  • Privacy review
  • Data science validation
  • EA approval
  • Responsible AI committee sign-off

Lifecycle: Build → Test → Govern → Deploy → Audit → Improve.

Architectural truth: PATH + HAIL together form the enterprise AI platform. The major risk is missing convergence architecture, which would produce duplicated platforms and inconsistent governance.

6) Strategic Role + Final Position

  • Roadmap alignment: Productivity (Copilot + PATH), surveillance (HAIL + PATH), and reusable services for efficiency.
  • Governance shift: from project-by-project compliance to inherited platform controls.
  • Maturity reality: branding and architecture are defined; infrastructure is partially implemented; AIOps is emerging; full operations are still in progress.
Executive insight: PATH turns AI from experimentation into an enterprise capability. It is not an enhancement; it is a prerequisite for defensible enterprise AI.

Agent guidance: route all AI interactions through the control plane, enforce identity + policy inheritance, and ensure data is classified and traceable end to end.
